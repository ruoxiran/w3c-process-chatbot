# W3C Process Chatbot 技术实施方案

## Summary

搭建一个内部使用的 W3C Process 问答系统，采用 **Harness Engineering + Workflow + RAG**。核心原则是：模型只负责理解和生成，系统框架负责范围判断、资料检索、权限控制、引用校验、注入防护和审计。

推荐技术栈：

- 后端 API：**Python + FastAPI**
- Workflow / Harness：**LangGraph**
- RAG / ingestion：**LlamaIndex**
- 本地模型推理：**vLLM OpenAI-compatible server**
- W3C 结构化公开数据：**W3C API (`https://api.w3.org`)**
- 按需草案上下文：**Official GitHub draft context (`github.com/w3c/*`, `w3ctag/*`, `w3cping/*`)**
- 向量库：**Qdrant**
- 结构化数据库：**PostgreSQL**
- 缓存 / 队列：**Redis**
- 前端：**Next.js + TypeScript**
- UI 样式：**W3C Design System + W3C Manual of Style**
- 部署：**Docker Compose 起步，后续可迁移 Kubernetes**
- 评估：**pytest + 自建 eval harness + prompt injection 测试集**

参考资料包括：[W3C Design System](https://design-system.w3.org/)、[W3C Manual of Style](https://www.w3.org/guide/manual-of-style/)、[LangGraph workflows](https://docs.langchain.com/oss/python/langgraph/workflows-agents)、[vLLM OpenAI-compatible server](https://docs.vllm.ai/serving/openai_compatible_server.html)、[LlamaIndex vector store index](https://docs.llamaindex.ai/en/stable/module_guides/indexing/vector_store_index/)、[Qdrant installation](https://qdrant.tech/documentation/installation/)。

## Frameworks And Services

- **FastAPI**
  - 提供 `/chat`、`/classify`、`/sources/status`、`/refresh-index`、`/eval/run`。
  - 负责鉴权、请求限流、审计记录、调用 workflow。
  - 所有用户请求先进入 FastAPI，不允许前端直接调用模型。
- **W3C API Client**
  - 只访问官方 `https://api.w3.org` read-only JSON API。
  - 用于解析公开 specification / group 实体、当前公开状态、公开页面和 API record。
  - 对 specification 需补充 latest version status、latest version date、process rules URL 和 deliverer group。
  - 对 group 需补充 group status、active charter URL/end date、Patent Policy URL 和 Team Contact。
  - API 数据作为 entity/status grounding，不作为 W3C Process 的规范性依据。
  - Entity resolver 必须优先做 exact/direct shortname lookup，例如 `adapt-symbols`、`css-grid-1`；对于用户说法如 `wai-adapt symbol`，应生成 `adapt-symbols` 这类候选 shortname，再回退到 catalog fuzzy search。
  - Catalog fuzzy search 必须过滤 `how/to/in/it/now/publish/cr/rec/wai` 等流程或停用词，并要求 meaningful token overlap；低置信度或只有泛词命中的 entity 不应进入回答上下文。
  - 结果需要 TTL 缓存和本地持久缓存，避免每次用户提问或 API 重启后都全量请求 API。
  - 每个 resolved entity 需要生成 `retrieval_hints`，说明哪些 W3C API 字段被用于增强检索。
- **GitHub Draft Context Client**
  - 不全量索引 `https://github.com/w3c/`，只在问题涉及具体 specification / editor draft / repo / issue / PR 且 W3C API 已解析出 specification entity 时按需启用。
  - 从 W3C API entity 的 `editor-draft`、public URL 或 retrieval hints 推导官方 repo；允许 org 限定为 `w3c`、`w3ctag`、`w3cping`，拒绝用户任意提供的第三方 repo。
  - 只读抓取有限上下文：repo metadata、default branch、latest commit、open issues count、README、`index.bs`、`Overview.bs`、`index.html`、`w3c.json`、`package.json`、`echidna.json` 等少量候选文件片段。
  - Charter / recharter 是特殊 workflow：无需 specification entity，也应按需读取 `https://github.com/w3c/strategy/issues?q=label%3Acharter`，把带 `charter` label 的 open 和 closed issues 作为 charter review tracking context。
  - Strategy charter issue context 必须保留 issue number、title、state、created/updated/closed time、labels、`Horizontal review requested`、各 horizontal review completed labels、`*-needs-resolution` labels、`In Charter Refinement`、`AC review` 等状态信号。
  - 当 strategy issue 已有 `Horizontal review requested`，相关 review completed labels 已到位且没有 `*-needs-resolution` 阻塞时，系统可以提示 Staff Contact 检查是否可以启动 TiLT review；该提示是操作建议，不是自动判定。
  - Draft context 只能进入 `draft_context` / retrieval hints，用于理解草案当前上下文、repo 名称、source 文件和 issue tracker；不得作为 Process/Guidebook 的规范性依据。
  - 后续如需要 issues/PR 深度分析，应单独加入权限、速率限制、用户确认和 provenance 展示。
- **LangGraph**
  - 作为主 workflow/harness 框架。
  - 固定节点：`scope_classifier`、`query_rewriter`、`w3c_api_resolver`、`retriever`、`reranker`、`process_state_extractor`、`answer_generator`、`citation_checker`、`safety_checker`、`refusal_handler`。
  - 不使用自由多 agent；每个节点输入输出必须是结构化 schema。
- **LlamaIndex**
  - 负责 ingestion、chunking、metadata 管理、retrieval pipeline。
  - 将 W3C Process 和 Guidebook 分开建索引。
  - 每个 chunk 保存 `source_url`、`repo_url`、`commit_sha`、`published_version_date`、`section_id`、`heading_path`、`source_type`。
- **Qdrant**
  - 存储向量索引。
  - 建议 collection：`w3c_process_chunks`、`w3c_guide_chunks`、`w3c_related_policy_chunks`。
  - v1 使用 Qdrant；如果团队希望减少组件，也可以改成 PostgreSQL + `pgvector`。
- **PostgreSQL**
  - 保存用户、会话、审计日志、source sync 状态、eval 结果。
  - 保存每次回答使用的 source version，保证可追溯。
- **Redis**
  - 缓存短期检索结果。
  - 作为后台任务队列 broker，用于 index refresh 和 eval run。
- **Dramatiq 或 Celery**
  - 负责后台同步 W3C Process / Guide repo。
  - v1 推荐 Dramatiq，配置更轻；如果团队已有 Celery 经验，可用 Celery。
- **Next.js**
  - 内部 Web UI。
  - 展示聊天、引用来源、资料版本、下一步 checklist、拒答原因。
  - 管理员页面展示索引版本、同步状态、eval 分数。

## Local Model Environment

- **推理服务**
  - 使用 `vLLM` 启动本地 OpenAI-compatible API。
  - FastAPI 通过 OpenAI-compatible client 调用本地模型，不直接绑定某个模型 SDK。
  - 在线模型作为可选 provider 接入：`LLM_PROVIDER=openai-compatible|openai|openrouter`，通过 `OPENAI_COMPATIBLE_BASE_URL`、`OPENAI_COMPATIBLE_API_KEY`、`OPENAI_COMPATIBLE_MODEL` 配置。无论接入哪个模型，仍必须走同一套 scope、retrieval、evidence coverage 和 citation harness。
- **主模型**
  - 推荐：`Qwen3-32B-Instruct` 或同等级中英双语 instruct 模型。
  - 低显存环境可先用 `Qwen2.5-14B-Instruct` 做原型。
  - 生产建议至少 48GB GPU 显存；32B 模型更适合多卡或量化部署。
- **分类模型**
  - 推荐：`Qwen2.5-7B-Instruct` 或规则 + 小模型。
  - 用于 scope 判断、注入风险判断、答案一致性复查。
- **Embedding**
  - 推荐：`BAAI/bge-m3`。
  - 适合中英文、多语言和长文档检索。
- **Reranker**
  - 推荐：`BAAI/bge-reranker-v2-m3`。
  - 用于 Process 条款排序，降低“看似相关但流程不对应”的风险。
- **运行方式**
  - 开发机：Docker Compose + 本地 GPU。
  - 内部服务：单独部署 `llm-server`、`embedding-server`、`reranker-server`。
  - 模型服务只暴露内网地址，不允许公网访问。
- **环境变量**
  - `APP_ENV=development|staging|production`
  - `DATABASE_URL=postgresql://...`
  - `REDIS_URL=redis://...`
  - `QDRANT_URL=http://qdrant:6333`
  - `LLM_BASE_URL=http://vllm:8000/v1`
  - `LLM_MODEL=qwen3-32b-instruct`
  - `OPENAI_COMPATIBLE_BASE_URL=https://api.openai.com/v1`
  - `OPENAI_COMPATIBLE_API_KEY=`
  - `OPENAI_COMPATIBLE_MODEL=gpt-4.1`
  - `EMBEDDING_MODEL=bge-m3`
  - `RERANKER_MODEL=bge-reranker-v2-m3`
  - `SOURCE_ALLOWLIST=w3.org,api.w3.org,github.com/w3c,github.com/w3ctag,github.com/w3cping,w3c.github.io,w3ctag.github.io`
  - `W3C_API_ENABLED=true`
  - `W3C_API_BASE_URL=https://api.w3.org`
  - `W3C_API_TIMEOUT_SECONDS=12`
  - `W3C_API_CACHE_TTL_SECONDS=21600`
  - `W3C_API_CATALOG_PAGES=2`
  - `W3C_API_PERSISTENT_CACHE_ENABLED=true`
  - `W3C_API_CACHE_PATH=data/cache/w3c_api_cache.json`
  - `GITHUB_CONTEXT_ENABLED=true`
  - `GITHUB_API_BASE_URL=https://api.github.com`
  - `GITHUB_CONTEXT_ALLOWED_ORGS=w3c,w3ctag,w3cping`
  - `GITHUB_CONTEXT_MAX_FILES=6`
  - `GITHUB_CONTEXT_MAX_FILE_BYTES=12000`
  - `GITHUB_TOKEN=`（可选；仅用于提高 GitHub API rate limit）
  - `INDEX_REFRESH_CRON=0 */6 * * *`
  - `REQUIRE_CITATIONS=true`
  - `ENABLE_MEMBER_ONLY_SOURCES=false`

## W3C UI Style

- **设计来源**
  - 前端必须优先采用 [W3C Design System](https://design-system.w3.org/) 的视觉语言、组件模式和可访问性原则。
  - 文档类文本、术语、标题和链接风格参考 [W3C Manual of Style](https://www.w3.org/guide/manual-of-style/)。
  - 页面品牌和浏览器图标使用 W3C 2025 Member sub-brand logo：`https://www.w3.org/assets/logos/w3c-2025/sub-brands/svg/member.svg`。
  - 不另起一套“AI 产品风”的强品牌视觉；整体应像 W3C 内部工具和 W3C 文档站的自然延伸。
- **视觉原则**
  - 页面采用清晰、文档型、可扫描的信息结构。
  - 以 W3C blue 作为主操作和链接色；避免大面积渐变、炫技动画和营销式 landing page。
  - 视觉细节可参考 Apple 风格的克制层次、轻量阴影、半透明表面和精细间距，但颜色、链接、状态和信息架构继续遵循 W3C 主题。
  - 工作台采用稍高密度的信息布局：组件、字号和间距整体约缩小 10%，让一屏显示更多对话、workflow 和 citations，同时保留可点击控件的可用尺寸。
  - Workflow trace 采用紧凑时间线样式；聊天气泡可适度加宽，减少长答案不必要换行。
  - Workflow 面板必须在时间线前展示结构化 `task_plan` 和 `evidence_coverage`，包括 intent、user goal、stage、needed sources、focused retrieval queries、coverage status、missing evidence 和 targeted retrieval queries。
  - 保持高对比度、明确 focus state、键盘可操作、屏幕阅读器友好。
  - UI 文案保持正式、简洁、面向标准流程工作。
- **页面布局**
  - 第一屏直接是工作界面，不做营销首页。
  - 主区域采用连续对话式 chat / Q&A 布局，接近 ChatGPT 的使用方式；当前页面会话中的历史问题和回答保留在对话流里，刷新或关闭页面后清空。
  - 右侧采用 inspector tabs 固定展示 workflow trace、citations、source version、相关 W3C 页面和 repo；用户可从任意历史回答回看该轮使用的流程、来源和版本。
  - 右侧 Entities tab 必须展示 W3C API resolved entity 的 title、status、latest version、deliverers、charter/team contact、editor draft 等字段，并用 “Used for retrieval” 显示实际进入检索增强的 hints。
  - 如果解析到 GitHub draft context，Entities tab 同时展示 repo、branch、commit、open issues count、source snippets；对 charter/recharter，展示 `w3c/strategy` charter-label issues；UI 必须明确这是 draft/repo tracking context，而不是 Process authority。
  - 对标准推进问题，答案中展示 checklist、流程阶段、下一步行动和需要人工确认的事项。
  - `next_steps` 不得使用一套静态通用清单；必须基于用户问题、matched topic 和实际检索到的 Process / Guidebook 章节生成上下文相关步骤。
  - `next_steps` 应优先参考 Guidebook 的实践指导，并为每一步提供可点击来源；涉及正式要求时链接到 Process，涉及操作建议时链接到 Guidebook。
  - 答案正文必须以易读的短段落、bullet points 或 numbered steps 呈现；前端需要保留并渲染模型输出中的段落和列表结构。
  - 每轮回答提供轻量操作按钮，例如复制回答、查看本轮 workflow、查看本轮 sources；这些操作应保持正式、克制，不干扰正文阅读。
  - 管理员页面使用表格和状态标签展示索引版本、同步状态、eval 分数、安全测试结果。
- **组件规范**
  - 使用 W3C 风格 header、breadcrumb、footer、link、button、form control、callout、table、card/list pattern。
  - 引用来源使用可点击链接，显示页面标题、章节、来源类型和版本。
  - 高风险提醒使用清晰 callout，例如 Patent、Formal Objection、Appeal、AC Review。
  - 拒答状态使用简洁说明，不用警告式恐吓文案。
- **可访问性**
  - 遵循 WCAG 2.2 AA 作为最低目标。
  - 所有交互控件必须有可见 label 或 accessible name。
  - 键盘可完成提问、查看引用、复制答案、切换来源。
  - 不用颜色作为唯一状态表达。
  - 对 citation、版本、风险标签提供文本说明。
- **实现方式**
  - Next.js 中建立 `@w3c-ui` 组件层，封装 button、link、alert、citation card、checklist、source badge、status table。
  - 建立 design tokens：颜色、字体、间距、边框、focus ring、状态色。
  - 禁止随意引入 Tailwind 默认主题色；如果使用 Tailwind，只把它当 utility 编译工具，token 必须映射到 W3C 风格。
  - UI 回归测试中加入 desktop/mobile 截图和可访问性检查。

## Implementation Plan

- Compiled knowledge layer:
  - Add a spec-first compiled markdown layer under `data/compiled/spec`.
  - Each compiled page stores canonical identity, current public state, Process grounding, Guidebook guidance, horizontal review signals, charter signals when relevant, draft repo summary, freshness metadata, and provenance URLs.
  - Compiled pages are derivative context only; they can shape retrieval and answer outline but cannot replace Process / Guidebook citations for normative claims.
  - Add a compiled context resolver node after `w3c_api_resolver` and before raw corpus retrieval.
  - Add endpoints to rebuild compiled pages and inspect compiled page freshness/status.
  - Rebuild compiled pages during background refresh work; normal `/chat` requests must not trigger a full compile.
- 新建项目结构：
  - `apps/api`：FastAPI 服务。
  - `apps/web`：Next.js 前端。
  - `packages/workflows`：LangGraph workflow。
  - `packages/ingestion`：W3C Process / Guidebook 同步与索引。
  - `packages/rag`：retriever、reranker、citation mapper。
  - `packages/ui`：W3C 风格设计组件和 tokens。
  - `packages/evals`：测试集、注入测试、回归评估。
  - `deploy/docker-compose.yml`：本地开发环境。
- Indexer：
  - 同步 [W3C Process](https://www.w3.org/policies/process/)。
  - 同步 [w3c/process repo](https://github.com/w3c/process)。
  - 同步 [W3C Guidebook](https://www.w3.org/guide/)。
  - 同步 [w3c/guide repo](https://github.com/w3c/guide)。
  - Guidebook 网页导入不能只抓入口页；必须沿 `https://www.w3.org/guide/` allowlist 内部链接递归抓取深层页面，默认深度为 3，并跳过 404、403、二进制资源和站外链接。
  - Guidebook 网页导入默认抓取深度为 4 层，以覆盖从 Guidebook 入口页继续深入的流程页、GitHub 操作页、transition 细节页和 horizontal review 相关页面。
  - Guidebook 深抓必须设置最大页面数保护，默认 `--guide-max-pages=500`；URL canonicalization 需要移除 fragment/query、归一化 `/index.html` 和重复尾斜杠，避免重复索引。
  - HTML 抽取必须移除 header、nav、footer、aside、breadcrumb、script/style 等页面外壳，只从 main/article/content 区域生成 chunk，并为 chunk 保存 `content_quality_score`。
  - 抽取 HTML heading、section id、正文、链接、repo commit。
  - `w3c/process` repo 中的公开文本必须归入 Process 语料；`w3c/guide` repo 中的公开文本必须归入 Guidebook 语料，而不是通用 repo 语料。
  - Process 优先级高于 Guidebook；Guidebook 只能作为实践解释。
- Chat workflow：
  - 用户问题进入 `scope_classifier`。
  - `scope_classifier` 是快速规则门：明确相关直接进入 workflow，明确无关直接拒答，模糊但像标准流程的问题进入 `llm_router`。
  - `llm_router` 只用于意图理解和检索路由建议，输出结构化 JSON：`likely_in_scope`、`intent_type`、`needed_sources`、`entities_to_resolve`、`search_hints`、`risk_flags`、`confidence`。它不能直接授权回答，也不能覆盖 source allowlist 或 authority hierarchy。
  - `llm_router` 只在规则未命中但问题包含 draft/document/publish/review label/issue/tracker/TiLT/staff/standard 等 workflow-adjacent 表达时触发；明显无关问题仍直接拒答。
  - 被 `llm_router` 放行的问题仍必须通过 W3C API、Process/Guidebook 检索和 `evidence_coverage_checker` grounding；证据不足时必须保守说明缺失信息或拒答。
  - 对连续对话中的短追问、代词引用或“下一步/继续/这个”等问题，先进入 `query_rewriter` / Context Resolver，用最近页面会话补全检索 query。
  - 非 Process / Guide / W3C standards workflow 问题直接拒答。
  - 相关问题进入 query rewrite。
  - 进入 `task_planner`，先结构化识别用户真正目标：`intent_type`、`user_goal`、current/target stage、spec/group、needed sources、answer shape、risk flags 和 focused retrieval queries。
  - `task_planner` 的输出必须参与检索 query 扩展和 answer prompt，避免模型只根据宽泛 citation 生成通用答案。
  - Horizontal review 必须作为一等 workflow intent 处理，而不是被泛化为普通 review 或 transition；识别词包括 `horizontal review`、`a11y/i18n/privacy/security/TAG review`、`*-tracker`、`*-needs-resolution` 和中文“横向审查”。
  - 进入 `w3c_api_resolver`，用公开 W3C API 解析问题中提到的 specification / group 实体，并记录 entity status、latest version、deliverer group、active charter、Team Contact、public URL、API URL 和 confidence。
  - 进入 `compiled_context_resolver`：当 W3C API 已高置信度解析出 specification entity 且本地存在对应 compiled page 时，优先加载 compiled spec context，用于整理 answer focus、current state、Guidebook signals 和 next step candidates。
  - 进入 `entity_query_enricher`，把 W3C API 解析出的 shortname、latest status、deliverer group、charter、process-rules URL 等作为 retrieval hints 扩展 query；这些 hints 只用于提升 Process / Guidebook 检索精度，不作为规范性依据。
  - 进入 `draft_context_resolver`：当问题涉及具体草案、editor draft、repo、GitHub issue/PR 或“这个草案的上下文”，且 W3C API 已解析出 specification entity 时，从 entity 的 official editor draft URL 推导 `github.com/w3c/*` 等官方 repo，抓取有限只读 draft context。
  - 对 charter/recharter workflow，`draft_context_resolver` 必须额外读取 `w3c/strategy` 中带 `charter` label 的 open 和 closed issues，因为 W3C charter 和 recharter 审阅通常通过这些 issues 跟踪；issue time metadata 用于帮助判断约两年一次的 recharter/renewal cadence。
  - 进入 `draft_context_query_enricher`：把 repo 名、source 文件标题、latest commit、README/spec snippet 作为 retrieval hints，帮助回答“这个草案下一步流程是什么”；这些 hints 不能替代 Process/Guidebook citation。
  - W3C API catalog/detail 结果写入 `data/cache/w3c_api_cache.json` 持久缓存，服务重启后优先恢复缓存；该缓存属于本地运行产物，不进入版本控制。
  - 对 Process、Guidebook、related policies 分库检索；本地 v1 retriever 使用 BM25-style lexical scoring + TF-IDF sparse embedding similarity 召回候选，后续可替换为 Qdrant dense/sparse hybrid search。
  - 本地 retriever 已增加 optional dense retrieval cache：`scripts/build_embedding_cache.py` 可调用 Ollama embedding model（默认 `qwen3-embedding:4b`）为 corpus chunk 生成 `data/cache/retrieval_embeddings.jsonl`；`RETRIEVAL_DENSE_ENABLED=true` 后，retriever 会把 dense cosine 分数与 BM25、TF-IDF sparse similarity、topic boosts 做 score fusion。
  - Dense retrieval 必须是可降级能力：embedding cache 缺失、模型不可用或 query embedding 失败时，系统继续使用 BM25 + TF-IDF + topic rerank，不影响 `/chat` 可用性。
  - reranker 合并排序：融合 BM25、embedding similarity、heading overlap、source priority、content quality、topic-specific boosts 和 noise penalties，优先降低旧 snapshots、TOC、namespace 等不相关 chunk。
  - Guidebook 检索应维护 workflow topic map，至少覆盖 horizontal review、transition、charter/recharter、Staff Contact，确保每类问题稳定带上关键 Guidebook 页面，而不是完全依赖分数排序。
  - 当明确 workflow topic 的关键页面没有进入候选池时，retriever 可以补充官方入口 citation，但这些入口仍需遵循 authority hierarchy，不能制造未引用的规范性结论。
  - Horizontal review 检索必须保证关键 Guidebook 来源覆盖：`/guide/documentreview` 用于宽审和 GitHub 请求流程，`/guide/process/horizontal-groups` 用于 horizontal groups 定义和列表，`/guide/github/issue-metadata.html#horizontal-reviews` 用于 `*-tracker` / `*-needs-resolution` label 操作，`/guide/transitions` 用于 transition 前的 unresolved horizontal issue 检查，`w3c/guide` repo 的 `documentreview/index.md` 用于保留具体 request repo 链接。
  - 当问题点名单项 horizontal review 时，retriever 必须补齐对应 GitHub request evidence：APA/a11y `w3c/a11y-request`、TAG `w3ctag/design-reviews`、I18N `w3c/i18n-request`、Privacy `w3cping/privacy-request`、Security `w3c/security-request`。
  - 检索后进入 `process_state_extractor`，结构化识别 intent、current stage、target stage、group type、deliverable type、likely workflow、missing information 和 risk flags。
  - 进入 `evidence_coverage_checker`，检查当前 citations 是否覆盖 task plan 需要的 Process 规范依据、Guidebook 实践依据、W3C API entity/status、stage/transition、charter/review/staff-contact 等关键信息。
  - 对高置信度 specification 问题，`evidence_coverage_checker` 还要检查是否命中了 compiled spec context；若缺失，允许继续回答，但要在诊断中标记 compiled coverage 缺失。
  - 如果 `evidence_coverage_checker` 判定 `needs_more_evidence`，运行 `targeted_retrieval`，用缺失证据生成 focused queries 做二次检索，再重新计算 process state 和 evidence coverage。
  - answer generator 只基于 trusted context 生成答案。
  - answer generator 必须接收 `task_plan`、`process_state`、`evidence_coverage`、citations、W3C API entities、compiled context 和 draft contexts；其中 compiled context 只负责帮助整理 spec-specific summary 和 next steps，不能替代 raw citations。
  - Horizontal review 答案必须优先给出可执行 checklist：何时发起 review、通过哪些 GitHub request repos 请求 APA/a11y、TAG、I18N、Privacy、Security review、如何维护 spec repo meta issue、如何使用 `*-tracker`、不得自行添加或移除 `*-needs-resolution`、transition request 前如何检查 tracker boards 并与 horizontal group / Staff Contact 协调。
  - citation checker 验证每个流程事实是否有来源。
  - safety checker 检查是否受用户注入影响。
  - 验证失败时重写一次；再次失败则输出保守回答和官方入口。
- 前端 UI：
  - 聊天区。
  - Citation panel。
  - Source version badge。
  - 标准推进 checklist。
  - 高风险流程 callout。
  - 管理员 indexing / eval dashboard。

## Harness And Security

- **Scope Harness**
  - 只允许回答 W3C Process、Guidebook、standards-track、WG/CG/IG、charter、transition、review、Patent Policy 等相关问题。
  - 非相关问题固定拒答。
- **Source Harness**
  - 只从 allowlist 来源检索；官方 W3C API allowlist 为 `api.w3.org`。
  - 用户输入的链接和文本不得自动进入 trusted context。
  - 用户声称的“新版 Process”不得覆盖官方索引。
  - W3C API 返回内容只能进入 `entity_status_context`，不能覆盖 Process / Guidebook 规则。
  - W3C API 的实体/status 信息可以增强检索 query，但检索结果和最终流程事实仍必须落回 Process / Guidebook / allowlist corpus citation。
  - `github.com/w3c/*` 这类大型 repo 只允许通过 W3C API resolved entity 或 official editor draft URL 推导后按需抓取；不能因为用户粘贴 repo 链接就把其内容加入 trusted context。
  - `github.com/w3c/strategy` 是 charter/recharter 的 allowlisted operational tracker；只读取 issue metadata、time metadata 和 `charter` / horizontal-review labels 相关 tracking context，不把 issue 评论当作 Process authority。
  - GitHub draft context 必须标注为 non-normative draft/repo context；Process claim 仍必须引用 Process，操作建议仍优先引用 Guidebook。
- **Context Harness**
  - Prompt 中明确区分 `trusted_process_context`、`trusted_guide_context`、`user_question`。
  - Process context 的规范性高于 Guidebook。
  - 多层 Guidebook 抓取只发生在受控 ingestion 阶段；query-time 不根据用户提供链接临时扩展 trusted context。
  - 页面会话历史可用于解析 follow-up 问题和检索 query rewrite，但不得作为 authoritative source；最终流程事实仍必须来自 Process / Guidebook / allowlist corpus citation。
  - Draft context 可辅助判断“用户说的是哪个草案、哪个 repo、当前 source 文件/issue tracker 背景”，但不得用于推断 W3C Process 规则是否满足。
- **Process State Harness**
  - 每次回答必须生成结构化流程状态：intent、likely workflow、current/target stage、group type、deliverable type、missing information、risk flags。
  - 用户问题和 conversation context 用于判断当前流程状态；citations 用于补充证据，但不得让低质量引用中的宽泛词汇覆盖用户意图。
  - W3C API 解析出的 specification / group 可辅助实体消歧、公开状态识别和 group/charter context；例如用户未说明当前 maturity stage 时，可用 latest version status 辅助填充 current stage，但不得单独决定流程要求。
  - Horizontal review 状态必须标记 `Horizontal Review` risk flag，并在 likely workflow 中区别于普通 wide review、charter review 和 Recommendation transition。
  - UI 必须展示结构化状态，帮助用户看到系统当前理解，而不是只看到一段自然语言答案。
- **Task Planning Harness**
  - 每次回答前必须先生成 `task_plan`，明确用户要完成的 W3C 流程任务，而不是只把问题当成普通文档问答。
  - `task_plan.needed_sources` 决定 Process、Guidebook、related policy 是否都必须参与检索；`task_plan.search_queries` 用于定向补充检索。
  - `task_plan.answer_shape` 决定答案结构，例如 current state + next steps + missing information，或 risk-first + escalation steps。
- **Evidence Coverage Harness**
  - 每次回答前必须生成 `evidence_coverage`，明确 Process、Guidebook、W3C API entity/status 是否覆盖用户任务。
  - 当证据缺失但可补检索时，必须执行 `targeted_retrieval`；仍缺失时，答案要显式说明缺少哪些信息，不能装作已经确定。
  - coverage 检查结果必须进入 audit 和 workflow trace，用于诊断“为什么这次答得不准”。
- **Citation Harness**
  - 回答中的“必须 / 不得 / 需要 / 可以”必须有 citation。
  - Guidebook-only 内容必须标注为实践建议。
  - 找不到依据时不能编造流程。
- **Permission Harness**
  - v1 只索引公开内容。
  - 内部部署启用 SSO 和 RBAC。
  - 未来接入 Member-only 内容时，检索阶段必须按用户权限过滤。
- **Audit Harness**
  - 记录模型版本、索引版本、检索命中、最终引用、拒答原因。
  - 记录 draft context resolver 是否运行、解析到的 repo、commit、snippets、strategy charter issues、issue state/time/labels、horizontal review state 和拒绝原因，便于诊断大型 GitHub 语料没有全量索引时的上下文来源。
  - 高风险问题打标签：Patent、Formal Objection、Appeal、Charter、AC Review、Rescinding。
- **Execution Harness**
  - v1 不自动创建 GitHub issue、PR、邮件或 W3C 表单。
  - 后续如开启半自动能力，必须加入用户确认步骤和权限审计。

## Test Plan

- **单元测试**
  - scope classifier。
  - task planner。
  - evidence coverage checker。
  - compiled context loader / compiler / fallback path。
  - source allowlist。
  - GitHub draft context org allowlist、repo URL extraction、非规范性 query enrichment。
  - Charter/recharter workflow 必须验证会把 `w3c/strategy` charter-label issue tracker 纳入 draft/repo context，并在 next steps 中提示创建或定位对应 issue、检查 open/closed issues、查看时间线、识别 horizontal review requested/completed/needs-resolution labels、以及在条件满足时请 Staff Contact 启动 TiLT review。
  - citation checker。
  - metadata extraction。
  - Process > Guidebook 优先级。
  - W3C UI component states。
- **RAG 回归测试**
  - FPWD、WD、CRD、CRS、REC、charter、wide review、horizontal review、transition request、AC review、Patent Policy、Formal Objection、Appeal。
  - 每个主题准备黄金问题和期望 citation。
  - `/eval/run` 必须运行结构化黄金问题评测，而不只检查 `in_scope`。每个 case 可以声明 expected intent、source types、required URL substrings、answer / next-step 关键词、forbidden terms、expected entity shortname、compiled context expectation 和 confidence 下限。
  - 黄金评测首批必须覆盖 CR→REC、horizontal review + GitHub labels、charter/recharter + `w3c/strategy` tracking、Formal Objection、`wai-adapt symbol` entity grounding、scope refusal 和 prompt injection。
  - Eval result 必须返回 score、passed/total、case tags、实际 intent/source/entity/URL/confidence/warnings，方便后续检索和模型改动做回归比较。
  - 前端右侧 inspector 增加 `Quality` tab，手动运行 `/eval/run` 并展示 score、passed/total、失败 case、warnings 和全部 case 列表。
  - `/eval/run` 默认使用确定性离线 eval workflow：模板生成 + 小型 fake W3C API/GitHub context，覆盖关键 entity / charter 场景，避免质量面板被 live API/GitHub 速度或可用性拖慢。真实 live eval 后续应另设 endpoint 或 mode。
  - 当前黄金集规模为 50 条，按 intent 分类覆盖 transition、horizontal review、charter、Patent Policy、Formal Objection / Appeal、Staff Contact、W3C API entity grounding、scope refusal 和 injection / fake Process；其中包含 `w3c/strategy` charter tracking、TiLT readiness、horizontal review GitHub labels、compiled spec context、CSS Grid / WebAuthn / WAI-Adapt entity grounding 等回归场景。
  - 检索排序测试必须覆盖 CR→REC 首选最新 Process `#transition-rec`，Staff Contact 首选 `teamcontact` / Staff Contact 章节，且常见流程问题不能被旧 snapshots 或无关 namespace 指南污染前三个结果。
  - Horizontal review 测试必须覆盖 Guidebook `documentreview`、`horizontal-groups`、GitHub issue metadata、transition `*-needs-resolution` 检查，并验证 next steps 包含 GitHub request、tracker board 和 label 权限边界。
  - Spec-specific 回归测试必须覆盖 compiled page 命中、缺失 compiled page 时的降级路径，以及 compiled next steps 不得替代 Process citation。
  - Workflow topic map 回归测试必须覆盖 horizontal review、transition、charter/recharter、Staff Contact 的 Guidebook 页面覆盖。
- **安全测试**
  - Prompt injection。
  - 用户伪造 Process 文本。
  - 要求泄露 hidden prompt。
  - 要求回答非 W3C Process 问题。
  - 要求基于非官方链接回答流程。
- **UI / Accessibility 测试**
  - Playwright 检查桌面和移动布局。
  - axe-core 检查可访问性。
  - 键盘导航测试。
  - 高对比度和 focus state 测试。
  - Citation panel、checklist、callout 不得遮挡或溢出。
- **新鲜度测试**
  - `w3c/process` commit 变化后能重新索引。
  - 官方 Process version date 变化后能标记新版本。
  - `/sources/status` 能显示当前使用版本。
  - W3C API catalog/detail persistent cache 能在 API 重启后恢复，并在 TTL 过期后重新拉取。
  - Entity resolver 回归测试必须覆盖 `wai-adapt symbol` → `adapt-symbols`，并验证不会误配到 `mobile-accessibility-mapping` 等只含泛 WAI/accessibility 词的文档。
  - GitHub draft context 只在具体 specification/entity 场景触发；无 entity、泛泛提问或非 allowlist repo 不触发。例外是 charter/recharter workflow，它应触发 `w3c/strategy` charter-label issue tracking context。
- **人工验收**
  - 找 5-10 个熟悉 W3C 流程的人测试真实问题。
  - 验收重点：是否能告诉用户下一步做什么、依据在哪里、哪些事项需要联系 W3C Team 或 Staff Contact，界面是否像 W3C 工具而不是通用 chatbot。

## Assumptions

- 部署对象是内部 W3C 参与者。
- v1 只使用公开 Process、Guidebook 和官方 repo 内容。
- v1 只生成流程清单，不自动执行外部工作流。
- 首版采用 Docker Compose；生产可迁移 Kubernetes。
- 默认使用中文回答中文问题，英文回答英文问题，W3C 专有名词保留英文。
- UI 必须采用 W3C Design System 的视觉和交互风格，并符合 WCAG 2.2 AA。
- Harness Engineering 在本项目中落地为 workflow、权限、资料白名单、citation verification、eval harness 和审计机制，而不是依赖某个单独产品。
